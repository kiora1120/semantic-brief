표준화의 봄: 시맨틱 레이어가
데이터 생태계의 공용어가 되다

Semantic Brief 편집팀 · 2026년 4월 15일

2026년의 데이터 업계를 관통하는 단 하나의 키워드를 꼽으라면, 그것은 단연 '표준화(standardization)'다. 지난 20년간 데이터 웨어하우스, 레이크, 메시(mesh) 등 수많은 패러다임이 등장했지만, 그 어떤 것도 '이 데이터가 무엇을 의미하는가'라는 근본 질문에 공통의 언어로 답하지 못했다. 각 조직은 저마다의 방언으로 비즈니스 로직을 코드에 새겨 넣었고, 그 결과 BI 도구마다, 분석가마다, 팀마다 서로 다른 숫자가 튀어나오는 '지표의 바벨탑'이 세워졌다. 이 고질적인 혼돈이 2026년, 마침내 구조적으로 해결될 조짐을 보이고 있다.

"데이터가 스스로의 의미를 선언할 수 있는 세계—그것이 바로 OSI가 향하는 방향이며, 온톨로지 공학이 수십 년간 꿈꿔온 비전이다."

그 중심에는 지난 1월 공개된 Open Semantic Interchange(OSI) v1.0 명세가 있다. Snowflake, dbt Labs, Salesforce, BlackRock, Databricks를 포함해 30개 이상의 조직이 Apache 2.0 라이선스로 공개한 이 명세는, 시맨틱 레이어의 핵심 구성요소—데이터셋, 지표(metric), 차원(dimension), 관계(relationship), 컨텍스트(context)—를 벤더 중립적인 포맷으로 정의한다. 마치 USB-C가 충전 규격을 통일하듯, OSI는 '지표의 의미'를 어떤 BI 도구에서든 동일하게 해석할 수 있도록 만드는 것을 목표로 한다. dbt Labs가 MetricFlow를 오픈소스화해 OSI의 참조 구현으로 기여한 것은 이 생태계가 단순한 선언에 그치지 않음을 보여준다.

이와 동시에, 가상 지식 그래프(Virtual Knowledge Graph) 진영에서도 의미 있는 움직임이 감지된다. Digital Science의 Ontopic 인수(2026년 3월)는 VKG 기술—데이터를 물리적으로 복제하지 않고 온톨로지 기반으로 통합·질의하는 방식—이 학술 연구의 영역을 넘어 엔터프라이즈 제품으로 본격 진입했음을 알린다. Ontopic의 Ontop 프레임워크는 기존 관계형 DB에 SPARQL 질의를 던질 수 있게 해주는 핵심 기술로, ETL 없이도 의미론적 데이터 통합을 가능하게 한다. 이는 OSI의 선언적 표준화와 방향이 다르지만 목적지는 같다: 데이터가 이동하지 않아도 의미가 전달되는 세계.

AI, 특히 LLM의 부상은 이 흐름을 한층 가속화하고 있다. GraphRAG가 단순한 임베딩 기반 검색의 한계를 극복하고 다중 홉(multi-hop) 추론을 가능하게 하면서, 지식 그래프의 구조적 표현력이 재평가받고 있다. ICLR 2026에서 선보인 LinearRAG는 기존 GraphRAG의 비용 장벽—LLM으로 관계를 추출하는 고가의 그래프 구성—을 경량 엔티티 추출만으로 대체해 실용성을 획기적으로 끌어올렸다. Netflix가 QCon London 2026에서 공개한 엔드-투-엔드 지식 그래프는 온톨로지가 대규모 관찰 가능성(observability) 인프라에 실제로 작동한다는 사실을 증명한다.

물론 낙관만 할 수는 없다. OSI가 진정한 표준이 되려면, 수십 년간 '오픈' 표준의 무덤이 된 시맨틱 웹의 역사를 되새길 필요가 있다. RDF, OWL, SPARQL은 기술적으로 탁월했지만 복잡성의 장벽에 막혀 주류가 되지 못했다. OSI가 성공하려면, 이론적 완결성보다 현장의 실용성을 우선해야 한다. 다행히 설계자들은 이를 인식하고 있는 듯하다. YAML 기반의 선언적 명세, SQL과의 호환성, 에이전틱 AI 시스템과의 연동—이 모든 것이 실무 데이터 팀이 오늘 당장 도입할 수 있는 경로를 염두에 두고 설계되었다. 시맨틱의 봄이 진짜 개화할지, 아니면 또 한 번의 이른 서리로 시들지—2026년 하반기가 그 답을 줄 것이다.

Technical Deep Dive

가상 지식 그래프(VKG)란 무엇인가:
ETL 없는 온톨로지 기반 데이터 통합의 모든 것

Digital Science의 Ontopic 인수로 VKG가 다시 주목받고 있다. 하지만 "가상 지식 그래프"란 정확히 무엇이고, 왜 지금 엔터프라이즈 아키텍처에서 의미가 있는가? 이 딥다이브에서 개념부터 구현까지 완전히 분해한다.

핵심 개념 정의

가상 지식 그래프(Virtual Knowledge Graph, VKG)는 데이터를 물리적으로 이동하거나 복제하지 않고, 기존 관계형 데이터베이스(또는 다른 데이터 소스) 위에 온톨로지 기반 의미론적 레이어를 씌워 RDF 트리플 또는 지식 그래프처럼 질의(query)할 수 있게 하는 아키텍처 패턴이다. OBDA(Ontology-Based Data Access)라고도 불린다.

VKG / OBDA 아키텍처 다이어그램

VKG의 핵심 작동 원리

VKG의 마법은 SPARQL-to-SQL 쿼리 리라이팅에 있다. 사용자가 SPARQL로 "모든 직원의 이름과 소속 부서를 알려줘"라고 질의하면, Ontop 같은 VKG 엔진은 R2RML 매핑 파일을 참조해 이를 최적화된 SQL로 변환해 실제 DB에 실행한다. RDF 트리플이 실제로 존재하지 않아도 된다—온톨로지 매핑이 실시간으로 가상(virtual) 트리플 구조를 제공하기 때문이다.

이것이 기존 지식 그래프 구축 방식—데이터를 ETL로 추출해 별도의 RDF 트리플스토어에 로드하는—과 근본적으로 다른 점이다. ETL 기반은 데이터 복제 비용, 신선도(freshness) 문제, 스키마 변경 시 전체 파이프라인 재구성이라는 삼중 부담을 안고 있다. VKG는 이 세 가지를 모두 제거한다.

핵심 표준: OWL 2 QL과 R2RML

VKG 구현에는 두 가지 W3C 표준이 핵심이다. OWL 2 QL은 데이터베이스 질의에 최적화된 온톨로지 언어로, 클래스 계층과 속성 관계를 정의한다. R2RML(RDB-to-RDF Mapping Language)은 관계형 DB의 테이블·컬럼을 RDF 트리플의 주어·서술어·목적어에 매핑하는 규칙을 기술한다.

◆ R2RML 매핑 예시: 직원 테이블 → RDF 트리플 

# employees 테이블을 ex:Employee 클래스에 매핑
@prefix rr: <http://www.w3.org/ns/r2rml#> .
@prefix ex: <http://example.org/ontology#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

<#EmployeeMapping>
  rr:logicalTable [ rr:tableName "employees" ] ;

  rr:subjectMap [
    rr:template "http://example.org/employee/{emp_id}" ;
    rr:class    ex:Employee ;
  ] ;

  rr:predicateObjectMap [
    rr:predicate ex:hasName ;
    rr:objectMap [ rr:column "full_name" ]
  ] ;

  rr:predicateObjectMap [
    rr:predicate ex:worksInDepartment ;
    rr:objectMap [
      rr:template "http://example.org/dept/{dept_id}"
    ]
  ] .

◆ SPARQL 질의 → Ontop이 SQL로 변환 

# 사용자가 작성하는 SPARQL 질의
PREFIX ex: <http://example.org/ontology#>

SELECT ?name ?deptName
WHERE {
  ?emp a ex:Employee ;
       ex:hasName           ?name ;
       ex:worksInDepartment  ?dept .
  ?dept ex:departmentName   ?deptName .
  FILTER(?deptName = "Engineering")
}

-- Ontop이 자동 생성하는 최적화 SQL
SELECT e.full_name AS name, d.dept_name AS deptName
FROM   employees e
JOIN   departments d ON e.dept_id = d.id
WHERE  d.dept_name = 'Engineering' ;

왜 지금 VKG인가: LLM 시대와의 접점

LLM 기반 Text-to-SQL이 주목받으면서, "자연어 → SQL"의 경로가 각광받고 있다. 하지만 LLM이 SQL을 생성하려면 스키마 구조를 이해해야 하고, 수백 개의 테이블이 있는 레거시 DB에서는 이것이 극도로 어렵다. VKG는 온톨로지 레이어를 통해 LLM에게 "비즈니스 언어로 된 데이터 지도"를 제공한다. "직원이 부서에서 일한다"는 의미론적 구조를 LLM이 이해하면, 실제 테이블 스키마의 복잡성은 숨겨진다. 이것이 VKG + LLM 조합의 실질적 가치다.

VKG vs. 전통적 ETL 기반 KG: 언제 무엇을 선택할까

VKG가 모든 상황에 적합한 것은 아니다. 의사결정의 핵심은 데이터의 이동 빈도와 쿼리 복잡도에 있다. 실시간 또는 nearly-real-time 데이터가 필요하고 원천 DB가 충분한 처리 능력을 보유하고 있다면, VKG는 탁월한 선택이다. 데이터가 이미 원천 시스템에 있고, 매핑 유지보수가 ETL 파이프라인 유지보수보다 단순하기 때문이다.

반면 복잡한 그래프 추론(OWL Full reasoning), 대규모 그래프 알고리즘(최단 경로, 커뮤니티 탐지), 또는 다양한 분석 쿼리 패턴이 필요한 경우에는 데이터를 물리적으로 트리플스토어(GraphDB, Stardog, Amazon Neptune 등)에 로드하는 전통적 KG가 적합하다. VKG는 SPARQL 질의를 SQL로 변환하므로, SQL이 표현하기 어려운 재귀적 그래프 순회에는 한계가 있다.

Ontop의 최신 기능과 에코시스템

오픈소스 Ontop(현재 v5.x)은 주요 RDBMS—PostgreSQL, MySQL, Oracle, SQL Server, Snowflake, Databricks—를 지원하며, Protégé 플러그인으로 온톨로지를 시각적으로 모델링하고 매핑을 검증할 수 있다. Ontopic Studio는 Ontop에 엔터프라이즈급 UI와 협업 기능, API 게이트웨이를 추가한 상용 제품이다. Digital Science 인수 이후 Ontopic Studio는 metaphactory 플랫폼과 통합돼, 온톨로지 관리·KG 탐색·SPARQL 엔드포인트가 단일 플랫폼으로 통합되는 방향으로 발전할 전망이다.

◆ Python으로 Ontop SPARQL 엔드포인트 질의하기 

import requests
from SPARQLWrapper import SPARQLWrapper, JSON

# Ontop이 노출하는 SPARQL 엔드포인트 (로컬 또는 Ontopic Studio)
ENDPOINT = "http://localhost:8080/sparql"

sparql = SPARQLWrapper(ENDPOINT)
sparql.setReturnFormat(JSON)

# 온톨로지 어휘로 질의—실제 DB 스키마를 몰라도 됨
sparql.setQuery("""
PREFIX ex: <http://example.org/ontology#>

SELECT ?empName ?salary ?managerName
WHERE {
  ?emp a ex:Employee ;
       ex:hasName    ?empName ;
       ex:hasSalary  ?salary ;
       ex:reportsTo  ?mgr .
  ?mgr ex:hasName    ?managerName .
  FILTER(?salary > 80000)
}
ORDER BY DESC(?salary)
LIMIT 20
""")

results = sparql.query().convert()

for row in results["results"]["bindings"]:
    print(
        row["empName"]["value"],
        "→ 매니저:", row["managerName"]["value"],
        "| 연봉:", row["salary"]["value"]
    )
# Ontop은 이 SPARQL을 내부적으로 최적화된 SQL JOIN으로 변환·실행함

VKG 도입 로드맵: 실무 팀을 위한 단계별 가이드

1단계 — 온톨로지 설계(2~4주): 비즈니스 도메인의 핵심 엔티티(클래스)와 관계(속성)를 OWL 2 QL로 정의한다. Protégé나 TopBraid Composer를 사용하면 시각적으로 모델링할 수 있다. 처음에는 3~5개의 핵심 엔티티로 시작해 점진적으로 확장한다.

2단계 — 매핑 작성(1~3주): R2RML 또는 Ontop Mapping Language(OBDA)로 DB 테이블을 온톨로지 클래스에 매핑한다. Ontopic Studio를 사용하면 GUI로 매핑을 생성하고 즉시 SPARQL로 검증할 수 있다.

3단계 — SPARQL 엔드포인트 노출(1주): Ontop을 Docker로 배포하거나 Ontopic Studio SaaS를 활용해 SPARQL 엔드포인트를 제공한다. GraphQL 또는 REST API 래퍼를 추가하면 프론트엔드 팀이 더 쉽게 소비할 수 있다.

4단계 — LLM 통합(선택): LLM에게 온톨로지 스키마(TBox)를 컨텍스트로 제공하면, 자연어 질의를 SPARQL로 변환하는 Text-to-SPARQL 시스템을 구축할 수 있다. LangChain의 GraphSPARQLQAChain 또는 커스텀 프롬프트로 구현 가능하다.
