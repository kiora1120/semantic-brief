엔터프라이즈 AI의 뼈대가 되는
지식 그래프의 부상

GraphRAG가 실험에서 프로덕션으로, Netflix의 E2E 온톨로지, Ontopic 인수까지 — 오늘의 시맨틱 인프라 총정리

Semantic Brief 편집팀 · 2026년 4월 3일

지식 그래프, 드디어 인프라의 중심에 서다 — 실험이 끝나고 프로덕션이 시작된다

지난 몇 년간 지식 그래프(Knowledge Graph)는 구글의 서치 패널이나 위키데이터 같은 거대 플랫폼의 전유물처럼 여겨졌다. 아니면 학계의 연구 주제이거나, 최소한 대규모 엔지니어링 팀을 보유한 빅테크의 사치쯤으로 분류됐다. 그런데 2026년 봄, 그 인식이 무너지고 있다. 넷플릭스가 QCon London에서 650TB 규모의 그래프 데이터를 밀리세컨드 단위로 처리하는 E2E 온톨로지 아키텍처를 공개했고, Digital Science는 VKG(가상 지식 그래프) 선구자인 Ontopic을 전격 인수했다. GraphRAG는 "이론적으로 좋은 기술"에서 기업 AI 인프라의 표준 컴포넌트로 자리를 옮겼다. 이것은 단순한 기술 트렌드의 부침이 아니다. 데이터 인프라 자체의 패러다임이 전환되는 순간이다.

왜 지금인가? 핵심은 LLM(거대언어모델)과의 결합이다. 벡터 임베딩 기반의 전통적 RAG는 문서 유사도 검색에는 탁월하지만, '누가 누구와 어떤 관계인가', '이 서비스 장애가 상위 시스템 어디에서 비롯됐는가' 같은 구조적·관계적 질문에 취약하다. 지식 그래프는 이 공백을 채운다. GraphRAG는 단순 검색 대비 정확도를 최대 3.4배 높인다는 벤치마크가 나왔고, Microsoft Research는 이미 이를 오픈소스로 공개해 생태계가 빠르게 확장 중이다.

"2026년은 지식 그래프가 PoC 단계를 벗어나 엔터프라이즈 AI 스택의 필수 레이어로 편입되는 원년이다. 벡터만으로는 부족하다는 현장의 목소리가 아키텍처를 바꾸고 있다."

넷플릭스의 사례는 특히 인상적이다. 그들이 구축한 E2E 지식 그래프는 단순한 데이터 카탈로그가 아니다. 사용자, 클라이언트, 서비스, 인프라까지 연결된 거대한 연결망 위에서 온톨로지가 일관된 타입과 관계를 정의하며, 장애 원인 추적(observability)을 자동화한다. 이 시스템은 "Knowledge Flywheel"이라는 사이클로 돌아가는데, 관찰하고(Observer), 풍부하게 만들고(Enrich), 추론하는(Infer) 세 단계를 반복하며 그래프 자체가 스스로 학습한다. AI와 온톨로지의 공생 구조다.

시맨틱 레이어 쪽에서도 조용한 혁명이 진행 중이다. dbt Semantic Layer는 2026년 1월 YAML 스펙을 대폭 단순화했다. measures가 metrics로 통합되고 별도의 semantic_models 키가 사라졌다. Snowflake Semantic Views는 3월 2일 표준 SQL 쿼리를 GA로 전환했다. 더 중요한 것은 Open Semantic Interchange(OSI)다. dbt Labs, Snowflake, Salesforce가 공동으로 추진하는 이 표준화 이니셔티브는 벤더 중립적 YAML 포맷으로 시맨틱 레이어 정의를 공유하는 것을 목표로 한다. 락인(lock-in)을 걱정하던 데이터팀들에게는 단비 같은 소식이다.

그러나 낙관론만 있는 것은 아니다. GraphRAG는 여전히 비용 문제를 안고 있다. 100만 토큰 코퍼스의 지식 그래프 구축에 20~50달러의 API 비용과 수 시간의 컴퓨트가 필요하다. 대규모 조직에서는 수백 달러가 순식간에 사라진다. 엔티티 인식 정확도도 도메인 특수성에 따라 60~85% 편차를 보인다. 지식 그래프가 인프라의 중심에 서려면, 구축과 유지 비용을 대폭 낮추는 도구와 워크플로우가 뒤따라야 한다. Ontopic과 같은 VKG 접근법은 기존 데이터를 변환 없이 그래프로 쿼리할 수 있어 이 비용 문제의 유력한 해법이 될 수 있다. Digital Science의 인수는 그 가능성에 대한 투자이기도 하다.

결론은 명확하다. 지식 그래프와 온톨로지는 더 이상 "나중에 검토할 기술"이 아니다. 넷플릭스가 보여줬고, 여러 인수합병이 증명했으며, 표준화 움직임이 시장의 성숙을 알리고 있다. 데이터 아키텍트와 AI 엔지니어라면 지금 당장 자신의 스택에 그래프 레이어를 어디에 넣을지 고민해야 할 때다.

GraphRAG 아키텍처 완전 해부 — 벡터 검색을 넘어 관계 추론으로

엔터프라이즈 AI 파이프라인에서 지식 그래프가 RAG를 어떻게 강화하는가

GraphRAG(Graph-enhanced Retrieval-Augmented Generation)는 전통적 벡터 기반 RAG의 구조적 한계를 극복하기 위해 지식 그래프를 검색 레이어에 통합한 아키텍처다. 2024년 Microsoft Research가 발표한 이후 급속히 발전해, 2026년 현재 다양한 변형과 최적화가 등장한 성숙한 패러다임이 됐다.

핵심 개념: 왜 그래프가 필요한가

전통적 RAG는 질의(query)를 벡터로 변환해 문서 청크와의 코사인 유사도로 관련 문서를 검색한다. 이 방식은 "X에 대해 설명하라"는 의미론적 질문에 효과적이지만, "A와 B의 관계는 무엇이며, C에게 어떤 영향을 미치는가"처럼 멀티홉(multi-hop) 관계 추론이 필요한 질문에서 취약하다. 지식 그래프는 엔티티(노드)와 관계(엣지)를 명시적으로 저장하므로, 구조화된 관계 탐색이 가능하다.

GraphRAG 파이프라인: 사용자 쿼리 → 엔티티 추출 + 벡터 검색 → 지식 그래프 탐색 → 컨텍스트 융합 → LLM 응답

파이프라인 단계별 분석

1단계 — 인덱싱 (오프라인): 문서 코퍼스에서 NER(개체명 인식)과 관계 추출을 통해 지식 그래프를 구축한다. 동시에 문서를 청크로 나눠 벡터 인덱스를 생성한다. 이 단계가 GraphRAG의 가장 비용이 큰 부분으로, 100만 토큰당 $20~50의 LLM API 비용이 발생한다.

2단계 — 쿼리 처리 (온라인): 사용자 쿼리에서 핵심 엔티티를 추출하고, 동시에 벡터 유사도 검색을 수행한다. 추출된 엔티티를 KG의 노드에 매핑하여 관련 서브그래프를 탐색한다.

3단계 — 컨텍스트 융합: KG 탐색으로 얻은 관계 경로와 벡터 검색으로 얻은 텍스트 청크를 통합한다. 이때 그래프 경로는 "A는 B의 자회사이며, B는 C의 파트너다"처럼 구조화된 사실을 제공하고, 벡터 청크는 맥락적 세부 사항을 보완한다.

4단계 — LLM 생성: 풍부해진 컨텍스트를 바탕으로 LLM이 최종 응답을 생성한다. 구조화된 관계 정보 덕분에 환각(hallucination)이 줄고 멀티홉 추론 정확도가 높아진다.

실제 구현 예시 (Python + NetworkX + LangChain)

GraphRAG 기본 구현 — KG 구축 및 쿼리
# pip install langchain networkx sentence-transformers
import networkx as nx
from langchain.graphs import NetworkxEntityGraph
from langchain_openai import ChatOpenAI
from langchain.chains import GraphQAChain

# 1. 지식 그래프 구축
graph = NetworkxEntityGraph()

# 엔티티 및 관계 추가 (실제로는 LLM 기반 추출)
graph.add_node("Ontopic", type="Company", domain="VKG")
graph.add_node("Digital Science", type="Company", domain="Research")
graph.add_node("Ontop", type="Software", license="Apache 2.0")

graph.add_edge("Digital Science", "Ontopic",
               relation="acquired", date="2026-03-25")
graph.add_edge("Ontopic", "Ontop",
               relation="develops", role="primary_maintainer")

# 2. GraphQA 체인 구성
llm = ChatOpenAI(model="gpt-4o", temperature=0)
chain = GraphQAChain.from_llm(
    llm,
    graph=graph,
    verbose=True
)

# 3. 관계 추론 쿼리
result = chain.invoke(
    "Digital Science가 인수한 회사가 개발하는 소프트웨어는 무엇인가?"
)
# → "Digital Science는 Ontopic을 인수했으며,
#    Ontopic은 Apache 2.0 라이선스의 Ontop을 개발합니다."

# 4. SPARQL 인터페이스 (Ontop을 통한 VKG 쿼리)
sparql_query = """
PREFIX ex: <http://example.org/>
SELECT ?software ?license WHERE {
  ex:DigitalScience ex:acquired ?company .
  ?company ex:develops ?software .
  ?software ex:license ?license .
}
"""
# Ontop이 이 SPARQL을 기존 SQL DB에 직접 변환하여 실행

2026년의 GraphRAG 발전 방향

LinearRAG(ICLR'26)처럼 엔티티-관계 추출 비용을 줄이는 경량화 연구와 함께, 멀티모달 지식 그래프(텍스트+이미지+표 통합), 동적 KG 업데이트(실시간 엔티티 추가), 그리고 표준화된 그래프 쿼리 인터페이스(GQL, SPARQL 2.0) 등이 주요 연구 방향으로 부상하고 있다. 특히 VKG 접근법은 기존 관계형 데이터베이스를 변환 없이 그래프로 노출하므로, KG 구축 비용 문제를 우회하는 현실적 대안으로 주목받는다.

GraphRAG vs 전통 RAG 성능 비교

실제 엔터프라이즈 벤치마크(Microsoft Research GraphRAG 공식 평가 기준)에서 GraphRAG는 전통 RAG 대비 다음과 같은 성능 개선을 보인다. 복잡한 멀티홉 질문(Complex multi-hop Q&A)에서 정확도 향상률 3.4×, 관계 추론(Relationship inference) 태스크에서 F1 스코어 +41%, 문서 요약 코히런스(Coherence) +22%다. 반면 단순 사실 검색(Factual retrieval)에서는 벡터 RAG와 유사하거나 약간 느리다 — 그래프 탐색 오버헤드 때문이다. 따라서 GraphRAG는 "모든 RAG를 대체"하는 것이 아니라, 관계 추론이 중요한 유즈케이스에서 벡터 검색을 보완하는 하이브리드 전략이 최적이다.

VKG (Ontop) SPARQL → SQL 매핑 예시 — OBDA 방식
# Ontop 매핑 파일 (.obda) 예시
# 기존 관계형 DB를 RDF 온톨로지로 매핑 (ETL 없음)

mappingId   MAPID-company
target      ex:{company_id} a ex:Company ;
               ex:name {company_name} ;
               ex:domain {domain} .
source      SELECT company_id, company_name, domain
            FROM companies

mappingId   MAPID-acquisition
target      ex:{acquirer_id} ex:acquired ex:{target_id} ;
               ex:acquisitionDate {acq_date} .
source      SELECT acquirer_id, target_id, acq_date
            FROM acquisitions WHERE status = 'completed'

# 위 매핑으로 아래 SPARQL이 자동으로 SQL로 변환됨
# SELECT ?acquirer ?target ?date WHERE {
#   ?acquirer ex:acquired ?target .
#   ?acquirer ex:domain "Research" .
#   ?acquirer ex:acquired ?target .
#   BIND(YEAR(?date) AS ?year) FILTER(?year = 2026)
# }
# → SELECT a.company_name, t.company_name, aq.acq_date
#     FROM companies a JOIN acquisitions aq ON a.company_id=aq.acquirer_id
#     JOIN companies t ON t.company_id=aq.target_id
#     WHERE a.domain='Research' AND YEAR(aq.acq_date)=2026
⚖️ 도구 비교

시맨틱 레이어 도구 선택은 팀 규모, 기존 스택, 거버넌스 요구사항에 따라 크게 달라진다. 2026년 기준 주요 플레이어들의 실무적 비교를 정리했다. 단순 기능 목록이 아닌, 실제 도입 시 체감하는 러닝커브와 비용 구조까지 반영했다.

📌 에디터 노트
OSI 표준화 동향

Open Semantic Interchange(OSI): 시맨틱 레이어 표준화의 게임 체인저

2025년 말 등장한 OSI(Open Semantic Interchange)는 경쟁사인 dbt Labs, Snowflake, Salesforce가 공동 추진하는 이례적인 표준화 이니셔티브다. 벤더 중립적 YAML 포맷으로 시맨틱 레이어 정의를 표준화하는 것이 목표이며, MetricFlow의 선언적 스펙을 핵심 레퍼런스로 채택했다.

OSI가 중요한 이유는 벤더 락인(vendor lock-in) 문제다. 현재 dbt Semantic Layer로 정의한 지표를 Cube나 Snowflake로 이식하려면 상당한 재작업이 필요하다. OSI가 완성되면 한 번 정의한 시맨틱 레이어가 지원 도구 전반에서 호환된다. 이는 데이터팀의 이식성과 협상력을 동시에 높인다.

물론 아직 초기 단계다. 모든 벤더가 완전히 채택하기까지는 시간이 걸릴 것이며, AtScale 같은 독점 플레이어들은 채택이 느릴 수 있다. 그러나 방향성은 명확하다 — 시맨틱 레이어는 특정 플랫폼의 기능이 아닌, 독립적인 표준 레이어로 진화하고 있다.

OSI의 기술적 핵심은 MetricFlow에서 발전한 선언적 YAML 스펙이다. 지표(metric), 차원(dimension), 엔티티(entity), 시맨틱 모델(semantic model)의 4가지 핵심 구성요소를 정의하며, 각 구성요소는 벤더별 구현과 독립적으로 표현된다. 예를 들어 metric: revenue는 Snowflake에서는 Semantic View로, dbt에서는 MetricFlow 정의로, Cube에서는 measure로 각각 구현되지만 OSI 레이어에서는 동일한 정의를 공유한다.

데이터팀 입장에서 OSI 채택의 가장 큰 실익은 "정의 한 번, 어디서나"다. 현재 많은 조직이 같은 비즈니스 지표를 각기 다른 BI 도구와 데이터 플랫폼에서 각각 정의·관리하면서 불일치와 거버넌스 문제를 겪는다. OSI가 안착하면 이 비효율이 구조적으로 해소된다. 2026년 중 주요 플랫폼들의 OSI 베타 지원이 예상되며, 2027년에는 본격적인 상호운용성 테스트가 시작될 것으로 전망된다.
