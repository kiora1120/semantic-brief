이번 주 플래그십 에세이

AI가 틀릴 때,
문제는 모델이 아니라 컨텍스트다
— 온톨로지가 LLM의 두뇌를 교정하는 법

Semantic Brief 편집팀 · 2026년 3월 30일 · 사설

대규모 언어 모델이 틀린 답을 내놓을 때, 우리는 즉각적으로 모델 자체를 의심한다. 더 큰 파라미터, 더 정교한 파인튜닝, 더 긴 컨텍스트 윈도우. 이것이 지난 4년간 AI 산업의 지배적 내러티브였다. 그러나 2026년의 시점에서 이 프레임은 근본적으로 재검토돼야 한다. 모델이 아니라 컨텍스트가 문제라는 인식이, 온톨로지와 지식그래프 기술의 르네상스를 이끌고 있다.

생각해보면 단순하다. 어떤 의사가 환자를 진찰할 때, 그 환자의 진료기록이 단편적이거나 모순되거나 구조화되지 않아 있다면 — 아무리 뛰어난 의사라도 오진할 가능성이 높아진다. 의사의 실력이 문제가 아니라, 그 의사에게 주어진 정보의 질이 문제다. LLM도 마찬가지다. GPT-4o든 Claude든 Gemini든, 입력 컨텍스트가 잘못 구성되어 있으면 모델은 그것을 바탕으로 그럴듯한 오답을 생성한다. 이것이 환각(hallucination)의 본질이다.

"환각은 모델의 결함이 아니다. 잘못 구성된 컨텍스트에 대한 합리적 반응이다. 온톨로지는 그 컨텍스트를 수술한다."
— Semantic Brief 편집팀

온톨로지가 하는 일을 명확히 짚어보자. 온톨로지는 단순한 분류 체계가 아니다. 도메인 내의 개념, 관계, 제약 조건을 기계가 읽을 수 있는 형식으로 명시화한 지식 구조다. "의약품은 최대 하나의 주성분을 가진다"는 제약을 OWL로 표현하면, 이 제약을 위반하는 정보가 RAG 파이프라인에 들어오더라도 시스템이 이를 필터링하거나 경고할 수 있다. 반면 벡터 데이터베이스는 이런 논리적 구조를 이해하지 못한다 — 유사도 점수만 있을 뿐이다.

핵심 개념: 온톨로지 기반 컨텍스트 교정의 3단계

1단계 — 구조화(Structuring): 도메인 지식을 OWL/RDF 형식으로 명시. 개념 간 계층 관계, 도메인/범위 제약, 공리(axiom)를 정의한다.

2단계 — 매핑(Mapping): R2RML 등의 표준 언어로 기존 관계형 DB와 온톨로지 레이어를 연결. 데이터를 이동시키지 않고 가상의 지식그래프(VKG)를 생성한다.

3단계 — 제약 추론(Reasoning): 검색 단계에서 온톨로지 제약을 필터로 활용. 논리적으로 불일치한 정보가 LLM에 전달되기 전에 차단한다.

이 흐름의 실질적 증거가 2026년 1분기에 집중적으로 등장하고 있다. Microsoft의 GraphRAG v2가 벡터 RAG 대비 3.4배의 정확도를 보고했고, ICLR 2026에 채택된 LinearRAG 논문은 KG 추론의 시간 복잡도를 O(n²)에서 O(n)으로 낮췄다. Gartner는 지식그래프를 생성형 AI의 "Critical Enabler"로 공식 분류했다. 그리고 Digital Science가 Ontop VKG 기술을 가진 Ontopic을 인수했다. 이 사건들은 개별적으로 일어난 게 아니다 — 하나의 패러다임 전환을 가리키는 동시다발적 신호들이다.

가장 주목할 부분은 이 흐름이 학계에서 산업계로, 연구에서 프로덕션으로 이동하고 있다는 점이다. Neo4j와 Stardog는 GraphRAG 파이프라인을 엔터프라이즈 제품에 통합했고, metaphactory는 VKG를 통해 ETL 없이 기존 DB를 지식그래프로 노출하는 OBDA 기능을 확장하고 있다. 금융권에서는 JPMorgan과 Goldman Sachs가 FIBO 온톨로지를 기반으로 규제 컴플라이언스를 자동화하기 시작했다.

그러나 우리는 과도한 낙관론을 경계해야 한다. 온톨로지 구축은 여전히 비용이 높고 유지보수가 어렵다. 도메인 전문가와 엔지니어의 긴밀한 협업이 필요하며, 온톨로지의 범위가 넓어질수록 추론 엔진의 성능 병목이 발생한다. R2RML 매핑은 강력하지만, 복잡한 집계 로직이나 비정형 데이터에는 한계가 있다. 온톨로지는 만능이 아니다 — 그것은 정밀 도구다.

"온톨로지의 가치는 모든 문제를 푸는 것이 아니라, 가장 중요한 오류 — 논리적 불일치와 관계 추론 실패 — 를 구조적으로 차단하는 데 있다."
— Semantic Brief 편집팀

결론적으로, AI 인프라의 다음 경쟁은 모델 파라미터 싸움이 아니라 컨텍스트 품질 경쟁이 될 것이다. 더 정확하고, 더 일관되며, 더 설명 가능한 컨텍스트를 구성할 수 있는 조직이 AI에서 지속적인 우위를 확보한다. 온톨로지와 지식그래프는 그 경쟁의 핵심 무기다. 이번 주 Semantic Brief는 그 무기를 실전에서 어떻게 다루는지를 다룬다.

왜 중요한가

학술 출판·데이터 인프라 기업 Digital Science가 Ontop VKG(Virtual Knowledge Graph) 기술을 개발해온 Ontopic을 인수했다. 표면상 조용한 M&A지만, 시맨틱 기술 커뮤니티에서는 이를 VKG 기술의 메인스트림 진입 신호로 읽는다.

Ontop의 핵심 가치는 OBDA(Ontology-Based Data Access)다. 기존 관계형 DB의 데이터를 Triple Store에 옮기지 않고도, R2RML 매핑 파일 하나로 SPARQL 엔드포인트를 생성한다. 쿼리는 실시간으로 SQL로 변환되어 원본 DB에 전달된다. 데이터를 이동하지 않고 지식그래프의 이점을 즉시 누린다는 아이디어는 혁신적이지만, 지금까지는 주로 학술 연구 환경에 머물렀다.

metaphactory와의 통합이 완료되면, 제약·바이오·문화유산 분야의 엔터프라이즈 사용자들이 VKG를 즉시 활용할 수 있다. 진입 장벽이 낮아지고, 시장이 커진다.

왜 중요한가

Snowflake Semantic View Autopilot이 GA(Generally Available)로 전환됐다. 데이터 엔지니어가 "월별 구독 매출 트렌드를 보여주는 뷰를 만들어줘"라고 입력하면, AI가 스키마를 분석해 집계 로직과 메트릭 정의를 포함한 시맨틱 뷰를 자동으로 생성한다.

표면적으로는 편의 기능처럼 보이지만, 진짜 의미는 시맨틱레이어 구축의 진입 장벽 파괴다. 지금까지 시맨틱레이어 설계는 데이터 모델링 전문가의 영역이었다. Autopilot은 이 전문성을 AI가 일부 대체함으로써, 더 많은 팀이 시맨틱레이어의 이점을 누릴 수 있는 길을 연다.

특히 GA와 함께 추가된 Row-Level Security 자동 상속은 보안팀의 우려를 해소하는 핵심 기능이다. Cortex Analyst의 LLM이 권한 밖 데이터를 절대 반환하지 않도록 기존 RLS 정책이 시맨틱 뷰 레이어에 자동 적용된다.

"Autopilot은 단순 자동화가 아니다 — 시맨틱레이어를 누가 만드는가라는 질문에 대한 답을 바꾸고 있다."

왜 중요한가

복수의 연구팀이 발표한 RAG 벤치마크에서 GraphRAG의 우위가 확인됐다. 3.4배라는 숫자는 단순 성능 개선이 아니라 패러다임 전환을 의미한다. 벡터 RAG는 의미적으로 유사한 청크를 잘 찾지만, "A가 B의 부서장이고 B가 C 프로젝트를 관리할 때, C의 최종 승인권자는 누구인가?" 같은 다중 홉 관계 추론에서 구조적 한계를 보인다.

GraphRAG는 엔티티 간 관계를 그래프 경로로 추론해 이 문제를 해결한다. Microsoft GraphRAG v2는 커뮤니티 요약을 통한 글로벌 추론과 지역적 세부 추론을 결합하는 하이브리드 접근으로 특히 강점을 보인다.

dbt MetricFlow 0.9가 릴리즈됐다. 가장 주목할 변화는 Derived Metric 중첩 참조가 완전히 지원된다는 점이다. 이제 하나의 메트릭이 다른 메트릭을 참조하는 복잡한 비즈니스 로직을 단일 YAML 정의로 표현할 수 있다. 이 기능은 엔터프라이즈 데이터 팀이 MetricFlow를 진지하게 도입하는 데 가장 큰 장애물 중 하나였다.

기술 딥다이브 · R2RML & VKG

R2RML 매핑: 관계형 DB를 지식그래프로 변환하는 기술 — ETL 없이 온톨로지 레이어를 쌓는 법

수백만 행의 관계형 데이터를 Triple Store로 복사하는 대신, 매핑 규칙 파일 하나로 SPARQL 쿼리가 실시간 SQL로 변환된다. Virtual Knowledge Graph(VKG)의 핵심 표준인 R2RML을 해부한다.

관계형 데이터베이스는 전 세계 데이터의 절대다수를 저장하고 있다. ERP, CRM, EMR, 금융 원장 — 모든 운영 데이터가 테이블과 외래 키로 구성된 구조 속에 있다. 온톨로지 기반 지식그래프가 이 방대한 데이터를 활용하려면 두 가지 선택지가 있었다. 첫째, 데이터를 전부 Triple Store로 복사한다(전통적 ETL 방식). 둘째, 원본 DB를 그대로 두고 온톨로지 레이어를 "가상으로" 올린다(VKG 방식). R2RML은 두 번째 선택지를 가능하게 하는 W3C 표준이다.

R2RML(RDB to RDF Mapping Language)은 관계형 DB의 테이블과 컬럼을 RDF 트리플로 매핑하는 규칙을 정의하는 언어다. 핵심 개념은 단순하다: "이 테이블의 이 컬럼 값을 이 RDF 속성으로 표현해라." 이 매핑 파일을 기반으로, SPARQL 쿼리 엔진(Ontop 등)은 들어오는 SPARQL 쿼리를 분석해 대응하는 SQL로 변환하고, 원본 DB에서 결과를 가져와 RDF 형식으로 반환한다.

R2RML 매핑 아키텍처 — 쿼리 변환 흐름

위 다이어그램이 보여주듯, 애플리케이션은 SPARQL 쿼리를 VKG 엔진에 보낸다. 엔진은 온톨로지 정의와 R2RML 매핑 파일을 참조해 동등한 SQL 쿼리를 생성하고, 원본 관계형 DB에서 데이터를 가져와 RDF 형식으로 변환해 반환한다. 데이터는 원본 DB를 단 한 번도 떠나지 않는다.

Turtle · R2RML
# R2RML 매핑 예시: drug_table → pharma 온톨로지
# 파일: mappings/drug-mapping.ttl

@prefix rr:    <http://www.w3.org/ns/r2rml#> .
@prefix ex:    <http://example.org/ontology/pharma#> .
@prefix xsd:   <http://www.w3.org/2001/XMLSchema#> .

:DrugTriplesMap
  rr:logicalTable  [ rr:tableName "drug_table" ] ;

  # Subject: 각 행의 IRI 생성
  rr:subjectMap [
    rr:template   "http://example.org/drug/{id}" ;
    rr:class      ex:Drug ;
  ] ;

  # Predicate-Object: 이름 매핑
  rr:predicateObjectMap [
    rr:predicate  ex:drugName ;
    rr:objectMap  [ rr:column "name" ; rr:language "ko" ] ;
  ] ;

  # Predicate-Object: 화학식 매핑
  rr:predicateObjectMap [
    rr:predicate  ex:molecularFormula ;
    rr:objectMap  [ rr:column "formula" ; rr:datatype xsd:string ] ;
  ] ;

  # Predicate-Object: 제조사 조인 (외래키)
  rr:predicateObjectMap [
    rr:predicate  ex:manufacturedBy ;
    rr:objectMap  [
      rr:template  "http://example.org/company/{manufacturer_id}" ;
    ] ;
  ] .

# 생성되는 트리플 예시:
# <http://example.org/drug/001> a ex:Drug ;
#   ex:drugName "아스피린"@ko ;
#   ex:molecularFormula "C9H8O4" ;
#   ex:manufacturedBy <http://example.org/company/bayer> .

위 매핑 파일이 Ontop 엔진에 로드되면, "아스피린을 제조하는 회사의 모든 의약품을 조회하라"는 SPARQL 쿼리가 다음과 같은 SQL로 자동 변환된다:

SPARQL → SQL 변환 예시
-- SPARQL 입력:
-- SELECT ?drug ?name WHERE {
--   ?drug ex:manufacturedBy ?company .
--   <http://example.org/drug/001> ex:manufacturedBy ?company .
--   ?drug ex:drugName ?name .
-- }

-- 자동 생성된 SQL (Ontop 엔진):
SELECT
  d2.id         AS drug_id,
  d2.name       AS drug_name
FROM
  drug_table d1
  JOIN drug_table d2 ON d1.manufacturer_id = d2.manufacturer_id
WHERE
  d1.id = '001'
  AND d1.id != d2.id ;

이것이 VKG의 핵심 마법이다. 사용자는 온톨로지의 개념과 관계로 쿼리를 작성하고, 시스템이 그것을 DB가 이해하는 언어로 번역한다. 온톨로지 엔지니어와 DB 관리자의 역할 분리가 명확해지고, 도메인 전문가가 SQL을 모르고도 복잡한 데이터 탐색을 할 수 있게 된다.

✓ R2RML 적합한 상황
  • 기존 운영 DB를 이동하지 않고 온톨로지 레이어가 필요할 때
  • 데이터가 실시간으로 변경되어 최신성이 중요할 때
  • 이종 DB(MSSQL, PostgreSQL, MySQL 혼합) 통합
  • ETL 파이프라인 유지비용을 없애고 싶을 때
  • 연구기관의 이종 데이터 소스 통합 (논문·임상·특허)
× 주의가 필요한 상황
  • 복잡한 집계(윈도우 함수, 중첩 집계)가 많은 경우
  • 비정형 데이터(텍스트, 이미지)가 주요 소스인 경우
  • 원본 DB 스키마가 자주 바뀌는 불안정한 환경
  • 초고속 OLAP 쿼리(밀리초 단위 응답)가 필요한 경우

dbt MetricFlow 0.9
코드 우선 · Git 네이티브 · 멀티 DWH
코드팀최적 선택
▾

MetricFlow는 시맨틱레이어를 코드 아티팩트로 취급한다. YAML 파일로 메트릭을 정의하고, Git으로 관리하며, CI/CD 파이프라인에서 테스트한다. 이 철학은 이미 dbt를 사용하는 데이터 엔지니어링 팀에게 자연스럽게 맞는다. v0.9에서 Derived Metric 중첩 참조가 완전 지원되면서, 복잡한 비즈니스 로직도 단일 YAML에 담을 수 있게 됐다.

Snowflake, BigQuery, DuckDB, Databricks 등 주요 DWH를 모두 지원하고, 완전 오픈소스(MIT)로 락인 걱정이 없다. 단점은 초기 설정과 YAML 스키마 학습에 시간이 필요하다는 것이다.

Cube (구 Cube.js)
API 우선 · JS/TS 정의 · BI 통합 강점
API팀최적 선택
▾

Cube는 시맨틱레이어를 API 레이어로 노출하는 데 강점이 있다. JavaScript/TypeScript로 큐브(cube)와 메트릭을 정의하고, REST/GraphQL/SQL API로 BI 도구와 커스텀 애플리케이션에 동시에 제공한다. 풀스택 개발팀이 직접 데이터 API를 관리할 때 자연스럽다.

엔터프라이즈 플랜은 캐싱, RBAC, 멀티테넌시를 지원하지만 비용이 증가한다. dbt와 함께 사용하는 팀도 많다 — dbt로 트랜스폼을 처리하고 Cube로 API를 제공하는 패턴이다.

AtScale
엔터프라이즈 BI · MDX 호환 · 대기업 특화
대기업레거시 BI 팀
▾

AtScale은 기존 BI 투자를 보호하면서 클라우드 DWH로 전환하려는 대기업을 위한 솔루션이다. MDX 쿼리(Excel, Power BI 레거시) 호환성이 핵심 차별화 요소다. SSAS(SQL Server Analysis Services)를 사용하던 금융·제조 기업의 클라우드 마이그레이션 경로로 자주 채택된다.

러닝커브가 높고 비용이 상당하지만, 레거시 BI 스택과의 호환성이 결정적으로 중요한 환경에서는 대안이 없다.

MDX 호환SSAS 마이그레이션엔터프라이즈 거버넌스멀티 클라우드
❄️
Snowflake Semantic Views
Autopilot GA · Cortex Analyst 네이티브 · LLM 친화
Snowflake팀빠른 시작
▾

Snowflake를 메인 DWH로 사용하는 팀에게 Semantic Views는 가장 낮은 진입 장벽을 제공한다. Autopilot GA로 자연어 설명만으로 시맨틱 뷰를 생성할 수 있고, Cortex Analyst와 네이티브로 연동되어 비즈니스 사용자의 자연어 쿼리를 즉시 지원한다.

RLS 자동 상속으로 보안도 강하다. 단점은 명확하다: Snowflake에 완전히 종속되며, Git 기반 버전 관리가 되지 않는다. 코드 리뷰와 변경 이력이 필요한 팀에게는 적합하지 않다.
